# `benten-errors` — Internals deep-dive

*Read-only audit. Rewritten 2026-05-14 against HEAD `8141b94` (post Phase-4-Foundation close — tag `phase-4-foundation-close` at `0ce98d0`).*

## 1. What this crate does (plain English)

`benten-errors` is the single, frozen list of every error the Benten engine is allowed to emit. Each error has a stable string name like `"E_CAP_DENIED"` or `"E_PLUGIN_DEVICE_ATTESTATION_FORGED"`. Those strings are the contract: TypeScript code catches them by class, operators dashboard on them, audit pipelines route on them, and the drift detector keeps them in sync between docs, Rust, and TS. If the engine wants to fire a new error, it has to add a variant *here*; if a variant is here it has to fire in production code (the drift detector enforces both directions).

The crate itself is *only* an enum with three accessor functions and one tiny helper for parsing 3-segment capability strings. It deliberately has zero workspace dependencies and uses `no_std + alloc` so it can sit at the very bottom of the dependency graph — every other Benten crate (`benten-core`, `benten-graph`, `benten-caps`, `benten-eval`, `benten-engine`, `benten-sync`, `benten-platform-foundation`, `benten-renderer-tauri`, the napi bindings) imports `ErrorCode` from here without dragging in any heavier Benten machinery. This positioning was the closure of SECURITY-POSTURE Compromise #3 at Phase-1 close (the catalog used to live inside `benten-core::error_code`, which forced an unwanted edge on every crate that just wanted the string identifiers).

**Key invariants pinned by the file's own structure:**

1. Every variant has matching arms in `as_static_str` and `from_str` (drift between them is a wire-compat break).
2. `from_str(s.as_str()) == s` for every named variant (round-trip is the catalog's stability claim).

5. The variant list is `#[non_exhaustive]` so a Phase-4-Meta / Phase-5+ catalog extension is a minor-version bump, not a breaking change.

## 4. Public API surface

Plain English: a typed enum + three string accessors + a parser + an edge-routing accessor + a one-off helper for `prefix:domain:action` capability strings.

In more detail:

- **`ErrorCode`** is the enum. 168 named throwable variants today; `#[non_exhaustive]`; the trailing `Unknown(String)` variant exists so a future server that emits a newer code doesn't crash an older client — the raw string survives losslessly through `as_str`.
- **`as_str(&self) -> &str`** (`src/lib.rs:1089`) — returns the stable catalog code (e.g. `"E_INV_CYCLE"`). For `Unknown(s)` returns the stored string; for named variants delegates through `as_static_str`. This is the workhorse accessor; consumer crates' `.code()` methods call it.
- **`as_static_str(&self) -> &'static str`** (`src/lib.rs:1110`) — same return shape but with a `'static` lifetime. For `Unknown(_)` returns the sentinel `"E_UNKNOWN"` because the owned-String payload can't be promoted to `'static` without leaking. Single source of truth for the 168-arm catalog match; `as_str` delegates here (per r6-err-10 — used to be duplicated; the duplication was the bug).
- **`code(&self) -> ErrorCode`** (`src/lib.rs:1332`) — identity accessor. Looks pointless until you see test code like `let err = ErrorCode::CapDenied; err.code()` — lots of Phase-2a tests bind an `ErrorCode` directly and then call `.code()` as if it were a typed error wrapper. This method exists purely so those sites compile without per-test wrapping; it just clones `self`.
- **`routed_edge_label(&self) -> Option<&'static str>`** (`src/lib.rs:1367`) — answers "if this error fires in the middle of a handler walk, which `ON_*` outgoing edge does the evaluator take?" Returns one of `Some("ON_DENIED")` (cap-denial family, now includes all 4 thin-client session codes + the consent-substitution discrimination cluster from R6-FP-A), `Some("ON_NOT_FOUND")` (registry-miss family), `Some("ON_CONFLICT")` (optimistic-concurrency family), `Some("ON_OUTPUT_LIMIT")` (lone variant: `InvSandboxOutput`, mirrors the SANDBOX primitive's edge surface), `Some("ON_ERROR")` (the runtime-failure catch-all), or `None` (registration-time / builder-time / resume-protocol / dev-tooling / plugin-install-validation failures that have no in-graph edge — they surface at their call site instead). Carries the most editorial commentary in the whole file because routing decisions are load-bearing for the engine's evaluator behaviour.
- **`from_str(s: &str) -> ErrorCode`** (`src/lib.rs:1757`) — parser. Mirrors `as_static_str` arm-for-arm; falls back to `ErrorCode::Unknown(s.to_string())` on unrecognized input so forward-compat parsing never panics.
- **`Display`** — delegates to `as_str`. So `format!("{}", code)` emits the catalog string.
- **`PartialEq<ErrorCode> for &str`** + the two siblings — exists so test code can write `assert_eq!(err.code(), "E_CAP_DENIED")` with either operand order, where `err.code()` may be either `&'static str` or `ErrorCode`. Catalog-consolidation convenience from Phase-2a.
- **`PHASE_2A_FIRING_CODES`** — the 13 Phase-2a codes that have firing sites (catalog + R3 catalog-coverage test consume the same const so the two tests can never drift).
- **`PHASE_2A_RESERVED_CODES`** — the 5 `HostError` slots reserved in Phase 2a; firing sites landed in Phase 3 sync (catalog documents them as "reserved — fires in Phase 3").
- **`CapString`** + **`parse_cap_string`** — a tiny 3-segment string parser that accepts `prefix:domain:action` and rejects lone-`*` with `Err(CapScopeLoneStarRejected)`. The doc-comment still calls it a "Phase 2a stub" pending a real parser at G4-A; flagged in §9 below.

## 8. Thin-engine + composable-graph philosophy check

This crate is one of the cleanest expressions of the thin-engine principle in the workspace. It does exactly one thing (own the stable string identifiers), depends on nothing, exports a flat API surface, and refuses to grow runtime behaviour. A few notes:

### Well-respected examples

- **Zero workspace deps, `no_std + alloc`** (`Cargo.toml` + `src/lib.rs:32-39`). The crate could trivially depend on `thiserror` or `strum` to mechanically derive `Display` / `as_str` / `from_str`. It deliberately doesn't. Every consumer crate carries its own `thiserror` ergonomics around the enum; this crate stays the irreducible nucleus.
- **`#[non_exhaustive]` on the enum** (`src/lib.rs:51-53`). Forces downstream `_ =>` arms; means catalog growth in later phases is minor-version-compatible. This is exactly the right call for a catalog at the root of the workspace.
- **`ALL_CATALOG_VARIANTS` derives its count via `.len()`** (`tests/stable_shape.rs:479`). Bumping the list and forgetting to bump a separate counter is impossible by construction.
- **`as_str` delegates to `as_static_str`** (`src/lib.rs:1089-1094`) per r6-err-10 (Phase-2a). The match-arm body lives in exactly one place. The duplication that used to exist was actively a source of drift bugs.
- **`routed_edge_label` is exhaustively named, not wildcard-defaulted** (`src/lib.rs:1367` onwards). The previous `_ => "ON_ERROR"` wildcard had misrouted `WriteConflict`; EH2 fixed that and the named match shape now requires explicit family assignment for every new variant.
- **Dual-tripwire catalog-coverage** (`tests/stable_shape.rs:804` `catalog_variant_count_matches_enum`) — exhaustive `match` over `ErrorCode` so a new enum variant without a list entry fails to compile, plus a runtime length-mismatch assertion. Phase-4-Foundation R6-FP-C closure of ec-r6r1-2; protects against the latent-variant class of bug that allowed 14 throwables to silently escape the round-trip list across Phase-2b through Phase-4-Foundation R5.
- **Subset-closure pins per R5 cohort** (`error_codes_g23_a_subset_closure.rs` + `error_codes_g23_b_subset_closure.rs` + `error_codes_g24_d_subset_closure.rs`) — §3.5g cross-language rule-mirror inverse-direction enforcement. Each cohort's typed-error registry array at `benten-platform-foundation/tests/common/*_fixtures.rs` is mirrored here so the Rust enum can't drift from the test-side registry.

### Heavy-handed / inelegant / flag-for-review

- **`PartialEq<ErrorCode> for str` + `for &str` + reverse** (`src/lib.rs:1990-2004`). These exist to make Phase-2a test ergonomics happy (`assert_eq!(err.code(), "E_CAP_DENIED")`). The `PartialEq<ErrorCode> for str` impl in particular is unusual — it crosses the bare-`str` / `&str` boundary, and the doc comment doesn't fully justify keeping both `&str` and `str` impls. A future cleanup could pin which test sites depend on which direction and drop the unused half; the surface is small but the asymmetry feels like leftover from a refactor.

- **`code(&self) -> ErrorCode` is an identity method** (`src/lib.rs:1332`). The doc comment is candid that this is a "Phase 2a dx-r1-add" so test code like `let err = ErrorCode::Foo; err.code()` compiles. It's not actually catalog API — it's test-shim API leaking into the public surface. Future cleanup could either (a) document it as test-helper-only, (b) move it into a `#[cfg(test)]` impl, or (c) just delete it and require the tests to use the value directly. None of these are urgent.

- **`parse_cap_string` is a "Phase 2a stub"** (`src/lib.rs:2024-2051`). The doc-comment explicitly says "Real parser lands in G4-A." That was Phase-2a R5 work; Phase 2a/2b/3/Phase-4-Foundation have all closed without surfacing whether the real parser ever absorbed this stub. Construction sites: `cap_string_format_shape_locked` and `cap_string_format_escape_hatch` tests consume it. If the engine's real cap parser lives elsewhere now, this stub may be dead-code-with-tests; if it's still the only parser, the doc comment needs an update. The stub also conflates `CapDenied` (a runtime denial code) with parse failure (`s.is_empty()` → `Err(ErrorCode::CapDenied)`). A parse failure is not a denial — these are different conditions that happen to share a catalog code today. A future cleanup could mint a dedicated `CapStringMalformed` variant; the current behaviour is workable but mildly hazardous because operator dashboards filtering on `E_CAP_DENIED` will count parse failures as denials.

- **`esc_vector_variant_names_canonical.rs` reads sibling-crate source files from disk** (`tests/esc_vector_variant_names_canonical.rs`). It uses `CARGO_MANIFEST_DIR + "../benten-eval/src/sandbox/escape_defenses.rs"` and grep-matches for variant name substrings. This works but couples a zero-deps crate's tests to the workspace layout. If the eval crate ever moves files, this test breaks silently (it'll panic with "must exist" rather than the failure surfacing as a clear architectural-shape claim). The test is doing the right thing — it's an architectural-shape pin — but the implementation choice is fragile. A cleaner alternative is to keep the architectural-shape pin in `benten-eval` itself (or in `tests/phase_3_workspace/`) and let `benten-errors` test only what's local. Flagging for future consolidation rather than immediate fix.

- **The 2051-LOC `lib.rs`** is the natural shape for a catalog (one variant + 3 match arms + 1 doc-comment per catalog entry), but at 2051 LOC (up from 1666 at Phase-3-close) it is past where future splits become attractive. A natural split would be one module per family (`cap.rs` / `sandbox.rs` / `sync.rs` / `wait.rs` / `plugin.rs` / `schema.rs` / `materializer.rs` / `thin_client.rs` / etc.), each exporting a slice of variants, with a single `lib.rs` aggregator. The friction is that `match` exhaustiveness is what makes the enum a tripwire — splitting variants across modules would require the match arms to also distribute, which is awkward in Rust. Probably correct to defer this split until after v1; the LOC is monotonic-growing across phases and the current shape will not scale to 300+ variants gracefully.

- **No `serde` integration**. The enum doesn't derive `Serialize` / `Deserialize`. Consumers serialize via the string form. This is intentional (the `no_std + alloc` + zero-deps stance) and aligns with the philosophy, but Phase-4-Meta plugin manifests + Phase-8 decentralized plugin discovery will need stable wire encodings for ErrorCode — at that point a `serde` feature-flag (off by default) would be the natural addition.

### Composable-graph framing

The crate is a leaf in the dependency graph and pure data; it doesn't itself express engine composition. But it earns its keep relative to commitment #1 (12 primitives irreducible) and commitment #16 (SANDBOX doesn't bypass other primitives) by being the *contract surface* the engine uses to refuse compositions that would violate those commitments — e.g. `E_SANDBOX_NESTED_DISPATCH_DENIED` is what makes "SANDBOX → CALL → SANDBOX cap-laundering" a typed refusal rather than a runtime degrade.
