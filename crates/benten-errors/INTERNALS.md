# `benten-errors` — Internals deep-dive

*Read-only audit. Written 2026-05-11 against HEAD `5cab780` (post-pre-tag-sweep mid pre-v1 cleanup night-shift).*

## 1. What this crate does (plain English)

`benten-errors` is the single, frozen list of every error the Benten engine is allowed to emit. Each error has a stable string name like `"E_CAP_DENIED"` or `"E_SYNC_HLC_DRIFT"`. Those strings are the contract: TypeScript code catches them by class, operators dashboard on them, audit pipelines route on them, and the drift detector keeps them in sync between docs, Rust, and TS. If the engine wants to fire a new error, it has to add a variant *here*; if a variant is here it has to fire in production code (the drift detector enforces both directions).

The crate itself is *only* an enum with three accessor functions and one tiny helper for parsing 3-segment capability strings. It deliberately has zero workspace dependencies and uses `no_std + alloc` so it can sit at the very bottom of the dependency graph — every other Benten crate (`benten-core`, `benten-graph`, `benten-caps`, `benten-eval`, `benten-engine`, `benten-sync`, the napi bindings) imports `ErrorCode` from here without dragging in any heavier Benten machinery. This positioning was the closure of SECURITY-POSTURE Compromise #3 at Phase-1 close (the catalog used to live inside `benten-core::error_code`, which forced an unwanted edge on every crate that just wanted the string identifiers).

## 3. Files inventory in `src/`

Only one file.

### `src/lib.rs` (1666 LOC)

Crate root. Contains the `ErrorCode` enum (118 named variants + `Unknown(String)` forward-compat fallback), the `PHASE_2A_FIRING_CODES` + `PHASE_2A_RESERVED_CODES` const slices used by the drift tests, the four `ErrorCode` accessor methods (`as_str` / `as_static_str` / `code` / `routed_edge_label`), the `from_str` parser, a `Display` impl, three `PartialEq` cross-type impls bridging `ErrorCode` ↔ `&str`, the `CapString` struct, and the `parse_cap_string` helper.

**Crate-level attributes:** `#![forbid(unsafe_code)]`, `#![warn(missing_docs)]`, `#![no_std]`. The `extern crate alloc` line is what brings `String` and `Vec` into scope.

**Public exports (everything in this file):**

- `pub enum ErrorCode` (with `#[non_exhaustive]` — R6b bp-17, forces downstream `_ =>` fallbacks)
- `pub const PHASE_2A_FIRING_CODES: &[ErrorCode]` (13 entries)
- `pub const PHASE_2A_RESERVED_CODES: &[ErrorCode]` (5 entries)
- `pub struct CapString` (4 fields: `prefix`, `domain`, `action`, `reserved_extension_namespace`)
- `pub fn parse_cap_string(s: &str) -> Result<CapString, ErrorCode>`
- `impl ErrorCode { as_str, as_static_str, code, routed_edge_label, from_str }`
- `impl core::fmt::Display for ErrorCode`
- `impl PartialEq<ErrorCode> for &str` + `impl PartialEq<&str> for ErrorCode` + `impl PartialEq<ErrorCode> for str`

**Key invariants pinned by the file's own structure:**

1. Every variant has matching arms in `as_static_str` and `from_str` (drift between them is a wire-compat break).
2. `from_str(s.as_str()) == s` for every named variant (round-trip is the catalog's stability claim).
3. `as_str` and `as_static_str` return the same string for every named variant; the only delta is the lifetime (the `Unknown` arm cannot promote its owned `String` payload to `'static`).
4. `routed_edge_label` is exhaustive — no `_ =>` wildcard. EH2 (Phase 2a R6) replaced the wildcard with named family groupings because the wildcard had silently misrouted `WriteConflict` to `ON_ERROR` when it should be `ON_CONFLICT`; the named match shape now prevents that class of bug from recurring on later variants.
5. The variant list is `#[non_exhaustive]` so a Phase-4/5/8 catalog extension is a minor-version bump, not a breaking change.

## 4. Public API surface

Plain English: a typed enum + three string accessors + a parser + an edge-routing accessor + a one-off helper for `prefix:domain:action` capability strings.

In more detail:

- **`ErrorCode`** is the enum. 118 named variants today; `#[non_exhaustive]`; the trailing `Unknown(String)` variant exists so a future server that emits a newer code doesn't crash an older client — the raw string survives losslessly through `as_str`.
- **`as_str(&self) -> &str`** — returns the stable catalog code (e.g. `"E_INV_CYCLE"`). For `Unknown(s)` returns the stored string; for named variants delegates through `as_static_str`. This is the workhorse accessor; consumer crates' `.code()` methods call it.
- **`as_static_str(&self) -> &'static str`** — same return shape but with a `'static` lifetime. For `Unknown(_)` returns the sentinel `"E_UNKNOWN"` because the owned-String payload can't be promoted to `'static` without leaking. Single source of truth for the 118-arm catalog match; `as_str` delegates here (per r6-err-10 — used to be duplicated; the duplication was the bug).
- **`code(&self) -> ErrorCode`** — identity accessor. Looks pointless until you see test code like `let err = ErrorCode::CapDenied; err.code()` — lots of Phase-2a tests bind an `ErrorCode` directly and then call `.code()` as if it were a typed error wrapper. This method exists purely so those sites compile without per-test wrapping; it just clones `self`.
- **`routed_edge_label(&self) -> Option<&'static str>`** — answers "if this error fires in the middle of a handler walk, which `ON_*` outgoing edge does the evaluator take?" Returns one of `Some("ON_DENIED")` (cap-denial family — 27 variants), `Some("ON_NOT_FOUND")` (registry-miss family), `Some("ON_CONFLICT")` (optimistic-concurrency family), `Some("ON_OUTPUT_LIMIT")` (lone variant: `InvSandboxOutput`, mirrors the SANDBOX primitive's edge surface), `Some("ON_ERROR")` (the runtime-failure catch-all), or `None` (registration-time / builder-time / resume-protocol / dev-tooling failures that have no in-graph edge — they surface at their call site instead). Carries the most editorial commentary in the whole file because routing decisions are load-bearing for the engine's evaluator behaviour.
- **`from_str(s: &str) -> ErrorCode`** — parser. Mirrors `as_static_str` arm-for-arm; falls back to `ErrorCode::Unknown(s.to_string())` on unrecognized input so forward-compat parsing never panics.
- **`Display`** — delegates to `as_str`. So `format!("{}", code)` emits the catalog string.
- **`PartialEq<ErrorCode> for &str`** + the two siblings — exists so test code can write `assert_eq!(err.code(), "E_CAP_DENIED")` with either operand order, where `err.code()` may be either `&'static str` or `ErrorCode`. Catalog-consolidation convenience from Phase-2a.
- **`PHASE_2A_FIRING_CODES`** — the 13 Phase-2a codes that have firing sites (catalog + R3 catalog-coverage test consume the same const so the two tests can never drift).
- **`PHASE_2A_RESERVED_CODES`** — the 5 `HostError` slots reserved in Phase 2a; firing sites land in Phase 3 sync (catalog documents them as "reserved — fires in Phase 3").
- **`CapString`** + **`parse_cap_string`** — a tiny 3-segment string parser that accepts `prefix:domain:action` and rejects lone-`*` with `Err(CapScopeLoneStarRejected)`. The doc-comment calls it a "Phase 2a stub" pending a real parser at G4-A; flagged in §7 below.

## 7. Thin-engine + composable-graph philosophy check

This crate is one of the cleanest expressions of the thin-engine principle in the workspace. It does exactly one thing (own the stable string identifiers), depends on nothing, exports a flat API surface, and refuses to grow runtime behaviour. A few notes:

### Well-respected examples

- **Zero workspace deps, `no_std + alloc`** (`Cargo.toml:23-28` + `src/lib.rs:32-39`). The crate could trivially depend on `thiserror` or `strum` to mechanically derive `Display` / `as_str` / `from_str`. It deliberately doesn't. Every consumer crate carries its own `thiserror` ergonomics around the enum; this crate stays the irreducible nucleus.
- **`#[non_exhaustive]` on the enum** (`src/lib.rs:50-53`). Forces downstream `_ =>` arms; means catalog growth in later phases is minor-version-compatible. This is exactly the right call for a catalog at the root of the workspace.
- **`ALL_CATALOG_VARIANTS` derives its count via `.len()`** (`tests/stable_shape.rs:336-338`). Bumping the list and forgetting to bump a separate counter is impossible by construction.
- **`as_str` delegates to `as_static_str`** (`src/lib.rs:885-890`) per r6-err-10 (Phase-2a). The match-arm body lives in exactly one place. The duplication that used to exist was actively a source of drift bugs.
- **`routed_edge_label` is exhaustively named, not wildcard-defaulted** (`src/lib.rs:1104-1418`). The previous `_ => "ON_ERROR"` wildcard had misrouted `WriteConflict`; EH2 fixed that and the comment at `src/lib.rs:1080-1098` carries the rationale. Future variants will fail to compile if they don't get an explicit family assignment — which is what we want.

### Heavy-handed / inelegant / flag-for-review

- **`PartialEq<ErrorCode> for str` + `for &str` + reverse** (`src/lib.rs:1605-1619`). These exist to make Phase-2a test ergonomics happy (`assert_eq!(err.code(), "E_CAP_DENIED")`). The `PartialEq<ErrorCode> for str` impl in particular is unusual — it crosses the bare-`str` / `&str` boundary, and the doc comment doesn't fully justify keeping both `&str` and `str` impls. A future cleanup could pin which test sites depend on which direction and drop the unused half; the surface is small but the asymmetry feels like leftover from a refactor.

- **`code(&self) -> ErrorCode` is an identity method** (`src/lib.rs:1068-1071`). The doc comment is candid that this is a "Phase 2a dx-r1-add" so test code like `let err = ErrorCode::Foo; err.code()` compiles. It's not actually catalog API — it's test-shim API leaking into the public surface. Future cleanup could either (a) document it as test-helper-only, (b) move it into a `#[cfg(test)]` impl, or (c) just delete it and require the tests to use the value directly. None of these are urgent.

- **`parse_cap_string` is a "Phase 2a stub"** (`src/lib.rs:1639-1666`). The doc-comment explicitly says "Real parser lands in G4-A." That was Phase-2a R5 work; if G4-A has shipped (which the Phase-2a / Phase-2b / Phase-3 retrospectives suggest is the case for cap-policy work), then either the real parser absorbed this stub or this stub is still load-bearing and the comment lies. Worth a 5-minute audit. Construction sites: the test `cap_string_format_shape_locked` and `cap_string_format_escape_hatch` consume it. If the engine's real cap parser lives elsewhere now, this stub may be dead-code-with-tests; if it's still the only parser, the doc comment needs an update.

  The stub also conflates `CapDenied` (a runtime denial code) with parse failure (`s.is_empty()` → `Err(ErrorCode::CapDenied)`). A parse failure is not a denial — these are different conditions that happen to share a catalog code today. A future cleanup could mint a dedicated `CapStringMalformed` variant; the current behaviour is workable but mildly hazardous because operator dashboards filtering on `E_CAP_DENIED` will count parse failures as denials.

- **`esc_vector_variant_names_canonical.rs` reads sibling-crate source files from disk** (`tests/esc_vector_variant_names_canonical.rs:74-127`). It uses `CARGO_MANIFEST_DIR + "../benten-eval/src/sandbox/escape_defenses.rs"` and grep-matches for variant name substrings. This works but couples a zero-deps crate's tests to the workspace layout. If the eval crate ever moves files, this test breaks silently (it'll panic with "must exist" rather than the failure surfacing as a clear architectural-shape claim). The test is doing the right thing — it's an architectural-shape pin — but the implementation choice is fragile. A cleaner alternative is to keep the architectural-shape pin in `benten-eval` itself (or in `tests/phase_3_workspace/`) and let `benten-errors` test only what's local. Flagging for future consolidation rather than immediate fix.

- **The 1666-LOC `lib.rs`** is the natural shape for a catalog (one variant + 3 match arms + 1 doc-comment per catalog entry), but at 1666 LOC it's getting near where future splits become attractive. A natural split would be one module per family (`cap.rs` / `sandbox.rs` / `sync.rs` / `wait.rs` / etc.), each exporting a slice of variants, with a single `lib.rs` aggregator. The friction is that `match` exhaustiveness is what makes the enum a tripwire — splitting variants across modules would require the match arms to also distribute, which is awkward in Rust. Probably correct to defer this split until after v1; flagged because the LOC is monotonic-growing and the current shape will not scale to 300+ variants.

- **No `serde` integration**. The enum doesn't derive `Serialize` / `Deserialize`. Consumers serialize via the string form. This is intentional (the `no_std + alloc` + zero-deps stance) and aligns with the philosophy, but Phase-4 plugin manifests + Phase-8 decentralized plugin discovery will need stable wire encodings for ErrorCode — at that point a `serde` feature-flag (off by default) would be the natural addition.

### Composable-graph framing

The crate is a leaf in the dependency graph and pure data; it doesn't itself express engine composition. But it earns its keep relative to commitment #1 (12 primitives irreducible) and commitment #16 (SANDBOX doesn't bypass other primitives) by being the *contract surface* the engine uses to refuse compositions that would violate those commitments — e.g. `E_SANDBOX_NESTED_DISPATCH_DENIED` is what makes "SANDBOX → CALL → SANDBOX cap-laundering" a typed refusal rather than a runtime degrade. Same for `E_RESERVED_HANDLER_NAMESPACE` (rejects user handlers in the `engine:typed:` prefix per commitment #16), `E_VIEW_STRATEGY_A_REFUSED` / `E_VIEW_STRATEGY_C_RESERVED` (refuses user views from claiming engine-only strategies per the IVM seam), and the 14 `InvXxx` codes that enforce the 14 structural invariants at registration time. The catalog is the engine's "what compositions are illegal" vocabulary; growing the catalog is how the engine teaches itself new ways to say no.

## 8. Phase 4-Foundation + Phase 4-Meta expectations

The crate will mint new variants for every new error class the engine learns to recognize. Specifically:

- **Phase 4-Foundation / pre-v1 cleanup window** (already mid-flight per CLAUDE.md status at HEAD `5cab780`): the Class B β `read_node_as` work landed at PR #184. If the read-path principal threading surfaces new denial conditions (e.g. plugin attempting to read outside its manifest envelope), expect 1-2 new `E_PLUGIN_READ_*` variants. The 4 `todo!()` stubs at `crates/benten-engine/src/engine_wait.rs:1011-1026` were named in CLAUDE.md item #18 as the β migration target; once they fire real code, the typed error surface they emit lands here.

- **Phase 4 — plugin manifest schema + per-plugin DID/UCAN + layered consent** (per CLAUDE.md baked-in #18). Each layer mints catalog entries:
  - Install-time manifest validation: `E_PLUGIN_MANIFEST_INVALID`, `E_PLUGIN_MANIFEST_UNSIGNED`, `E_PLUGIN_MANIFEST_REQUIRES_EXCEEDS_USER_GRANT`.
  - Runtime delegation envelope check: `E_PLUGIN_SHARE_POLICY_VIOLATED`, `E_PLUGIN_CAP_NOT_TRACED_TO_USER_ROOT`.
  - Private-namespace cross-plugin refusal: `E_PLUGIN_PRIVATE_NAMESPACE_LEAK`.
  These don't exist today; they'll appear during Phase-4 R1 / R5 dispatches and bump the `CATALOG_VARIANT_COUNT` ledger.

- **Admin UI v0 / schema rendering** (likely a Phase-3.5 sub-surface): probably mints 1-3 codes around schema validation failures at the UI/engine boundary (`E_SCHEMA_RENDER_FAILED`, `E_SCHEMA_BIND_REJECTED`). These are speculative — no fire site exists today.

- **v1-assessment-window** (per CLAUDE.md baked-in #15): the missing_docs sweep is named as a v1-gate cleanup; this crate has `#![warn(missing_docs)]` and every public item is currently documented, so it's already compliant. The wasmtime Component-Model re-evaluation, identity-recovery protocol, and engine impl-block generic-cascade lift won't directly mint catalog entries for this crate but may mint entries in `benten-eval` / `benten-id` / `benten-engine` that route here.

The natural growth model is one catalog bump per substantive new typed denial surface; the file already has 11 inline "Phase-3 G_X wave-_y closes" paragraphs in `stable_shape.rs`'s variant-count ledger documenting that growth. Phase-4 will add a similar paragraph stack.

## 9. Open questions / unresolved internals

1. **`parse_cap_string` doc-comment still says "Phase 2a stub" + "Real parser lands in G4-A".** G4-A is well past landed (Phase-2a closed at tag `phase-2a-close`). Either the stub is actually fine and the comment is stale, or there's a real parser elsewhere now and this is dead code. Quick audit: check `crates/benten-caps/src/` for a `parse_cap_string` or similar. If a real one exists, this stub should either be deleted or its doc comment updated to "minimal parser used by the cap-string presence test only".

2. **`parse_cap_string` returns `Err(ErrorCode::CapDenied)` for empty/malformed inputs.** Conflates parse-failure-shape with capability-denial-runtime. Not currently a bug because nothing depends on the discrimination, but operator dashboards filtering on `E_CAP_DENIED` will count parse failures as denials. A dedicated `E_CAP_STRING_MALFORMED` variant would be cleaner; flagging for the v1-gate sweep.

3. **`PartialEq<ErrorCode> for str` (bare `str`, not just `&str`)** at `src/lib.rs:1615`. The other two impls (`for &str`, `for ErrorCode`) cover the common ergonomic cases; the bare-`str` impl handles `*self == other.as_str()` indirection. It's possible no test actually depends on the bare-`str` direction and the impl is leftover scaffolding. A grep on call sites would tell; flagging because the asymmetry-without-rationale is mildly confusing on first read.

4. **Anti-rename guards live mostly in `sandbox_codes_present.rs` and `wait_ttl_codes_present.rs` but not in `stable_shape.rs`**. CLAUDE.md non-negotiable rule #5 (no deprecated aliases) suggests every variant *rename* in the catalog's history should have an anti-rename guard. Today only SANDBOX (D19 rename + cr-g7a-mr-2 cleanup) and WAIT_TTL (Phase-3 G20-A2) carry these explicitly. If any other variants have been renamed across the catalog's history, those rename-anti-alias guards are missing. A 30-minute grep over `docs/history/PHASE-*.md` for "renamed" + "ErrorCode" would surface gaps.

5. **No serde support, no JSON wire-form derive, no proptest strategy.** These are conscious omissions today, but Phase-4 (plugin manifests serialize ErrorCode for runtime denial envelopes) and Phase-8 (decentralized plugin discovery — manifests are content-addressed + signed) will likely need at least a `serde` feature-flag. The crate's `default-features = []` posture is already set up for this; adding `[features] serde = ["dep:serde"]` would be the minimum invasive shape.

6. **`esc_vector_variant_names_canonical.rs` reads `../benten-eval/src/sandbox/escape_defenses.rs` from disk via filesystem path.** Architecturally fragile; if files move, the test panics rather than fails informatively. Worth relocating to `tests/phase_3_workspace/` or `tests/integration/` where cross-crate visibility is natural. Not urgent — the test works today — but flagging because the location violates the crate's otherwise-clean leaf-position.

7. **No `#[derive(Hash)]` on `ErrorCode`.** `Debug + Clone + PartialEq + Eq` are derived. Consumers wanting to use `ErrorCode` as a `HashMap` key today have to hash via `.as_str()`. Possibly intentional (the `Unknown(String)` variant would complicate `Hash` semantics), possibly oversight. Low priority.

8. **Variant ordering in the enum is roughly chronological-by-introduction, NOT alphabetical or family-grouped.** The match arms in `as_static_str` / `from_str` follow the same order. Readability of the file is somewhat hurt by this — finding "all cap-related variants" requires scanning the whole file because they're interspersed with SANDBOX, WAIT, SUBSCRIBE additions made later. A pre-v1 cleanup could reshuffle into family groups (with a clear migration commit so blame is preserved). Defer-OK; the variant *names* + *strings* are stable so the reshuffle is purely internal.
