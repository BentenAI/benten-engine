# benten-caps ‚Äî Internals

A plain-English deep-dive on the `benten-caps` crate. Companion to the public `lib.rs` docs; written for fresh agents coming into Phase 4-Foundation / Phase 4-Meta / v1-assessment-window work. Treats *why* and *what's load-bearing* as first-class.

Last refreshed: 2026-05-14 against main HEAD `8141b94` post Phase-4-Foundation R6-FP cluster admin-merge.

---

## 1. What this crate does

`benten-caps` is the **capability-policy layer** for the Benten engine. It owns the pre-write (and pre-read) hook trait ‚Äî `CapabilityPolicy` ‚Äî that the transaction primitive consults at commit time, plus the surrounding types every concrete policy backend needs: `WriteContext`, `ReadContext`, `PendingOp`, `CapabilityGrant`, `GrantScope`, the segment-wise attenuation check, the typed `CapError` family that maps 1:1 to stable codes in `docs/ERROR-CATALOG.md`, and a separate `RateLimitPolicy` trait whose plug-in shape composes alongside the capability policy. The crate ships three concrete `CapabilityPolicy` impls: `NoAuthBackend` (the zero-cost Phase-1 default that permits everything), `GrantBackedPolicy` (reads `system:CapabilityGrant` / `system:CapabilityRevocation` Nodes through a small `GrantReader` trait and denies writes whose scope has no unrevoked grant; principal-aware on the read side since Phase 4-Foundation R1 cap-r1-2 / cap-r1-10), and `UcanGroundedPolicy` (composes `GrantBackedPolicy` with `UCANBackend` proof-chain validation for the `cap:typed:*` namespace, audience-bound for cross-atrium replay defense). The durable UCAN backend itself ‚Äî `UCANBackend<B: GraphBackend>` ‚Äî lives at `src/backends/ucan.rs` and is native-only.

Phase 4-Foundation also added three native-only **plugin-trust-model** modules ‚Äî `plugin_delegation` (Layer 3 single-step gate), `manifest_scope` (G27-D manifest-aware scope derivation + audience-side envelope check), and `manifest_envelope_chain_validation` (G24-D-FP-2 full-chain Layer-2-‚Üî-Layer-3 walker per CLAUDE.md baked-in #18 three-layer trust model). These compose on top of the existing policy surface; they do NOT introduce a fourth `CapabilityPolicy` impl.

Per CLAUDE.md baked-in #7, the capability system is *pluggable policy*, not a fixed UCAN implementation. The trait is the contract; backends slot in. UCAN-specific identity types (DIDs, keypairs, signed UCAN envelopes, the in-memory chain-walker) live in the `benten-id` crate; this crate consumes them at the `[target.'cfg(not(target_arch = "wasm32"))'.dependencies]` boundary because identity work is full-peer-only per baked-in #17. The thin-client wasm32 build sees only the policy trait + `NoAuthBackend` + `GrantBackedPolicy` + `LegacyUcanStubBackend` + the typed `CapError` surface. The G27-D `manifest_scope` + G24-D `plugin_delegation` + G24-D-FP-2 `manifest_envelope_chain_validation` modules are also native-only ‚Äî they consume `benten-platform-foundation::PluginManifest` + `benten_id::Did`, and the runtime delegation check is shape (a) full-peer concern only (shape b/c read views go through `Engine::read_node_as` per CLAUDE.md #18 Class B Œ≤, not through delegation walks).

## 2. Dependency chain

**Workspace-in (the crate's own deps):**

- `benten-core` ‚Äî for `Cid`, `Node`, `Value`, `CoreError`, `WriteAuthority` (re-exported through `policy::WriteAuthority` as the single canonical type across core / graph / caps).
- `benten-errors` ‚Äî for the stable `ErrorCode` enum that `CapError::code()` maps into. The durable UCAN backend keys grants + revocations into the inherited KV surface (`g14b:grant:*`, `g14b:revoked:*`). *(COLLAPSE-WITH-RESIDUAL refinement-audit-2026-05 S3 P1: the `g14b:dev_revoke:*` device-revocation store was DELETED ‚Äî see `docs/SECURITY-POSTURE.md` Compromise #23 SUPERSEDED-BY-COLLAPSE; revocation is now the single `g14b:revoked:<ucan_cid>` user-root marker.)* Promoted from dev-dep to non-dev-dep at G14-B wave-4b.
- `benten-id` (native-only) ‚Äî for `Ucan`, `UcanError`, `Did`, `validate_chain_at` / `validate_chain_for_audience`. *(COLLAPSE-WITH-RESIDUAL S3 P1: the `DeviceRevocation` import + `validate_chain_with_device_revocations` were DELETED with the device-revocation pipe.)* Phase 4-Foundation G27-C ships a sibling `GrantReader` trait at `benten_id::grant_reader::GrantReader` with a CID-keyed companion method (`has_unrevoked_grant_for_grant_cid`); the trait LIVES in `benten-id` (not this crate) per arch-r1-10 ‚Äî splitting it sideways preserved this crate's dep-direction and let `benten-id`-only consumers query the grant store without taking a `benten-caps` dep. Same-named base methods compose via UFCS at call sites.
- `benten-platform-foundation` (native-only, prod edge added at G27-D / batch-3 ‚Äî Cargo-cycle-safe: foundation's reverse edge stays dev-only) ‚Äî for `PluginManifest`, `SharesPolicy`, `SharesPolicyDefault`. **External in:**

- `thiserror` ‚Äî the typed `CapError` derive.
- `blake3` ‚Äî for the BLAKE3 hash that computes the content-CID of a UCAN envelope (matching the engine's `Node::cid` scheme).
- `serde_ipld_dagcbor` ‚Äî for the DAG-CBOR encode/decode of UCAN bodies + device-revocation entries into the KV store.
- `subtle` ‚Äî pulled at this layer to pin the same minor version that flows through `benten-id`; any cap-side security-decision compare (signature equality, DID equality) goes through `ConstantTimeEq` per `crypto-major-4`.

**Dev-deps:** `proptest`, `tempfile`, `criterion`, and `benten-engine` (with `features = ["test-helpers"]`) for the few integration tests that need a full engine fixture (e.g. `toctou_iteration.rs`, `check_write_called_at_commit.rs`).

**Consumers out:**

- `benten-eval` consumes the trait + `WriteContext` / `ReadContext` to call `check_write` at transaction commit + `check_read` on the engine's read path.
- `benten-engine` is the composing layer: `EngineBuilder` owns the choice of which `CapabilityPolicy` to plug in (`NoAuthBackend` default, `GrantBackedPolicy` for the Phase-2b grant-reader path, `UcanGroundedPolicy` for the Phase-3 durable-UCAN path via the `capability_policy_ucan_durable` builder). The engine implements `GrantReader` against its own `RedbBackend` and injects it as `Arc<dyn GrantReader>` into `GrantBackedPolicy::new`. Phase-3 PR #199 shipped `Engine::revoke_capability_by_grant_cid` (resolves grant CID ‚Üí Node ‚Üí typed `scope` property ‚Üí routes through the existing `(actor, scope)` revoke path) ‚Äî closing the ¬ß13.11 namespace-mismatch silent fail-OPEN; the `benten-caps` surface is unchanged by that fix.
- `benten-platform-foundation` consumes the plugin-trust composition (G24-D `plugin_delegation` + G24-D-FP-2 `manifest_envelope_chain_validation` + G27-D `manifest_scope`) via integration tests that drive `validate_chain_with_manifest_envelope` + `check_scope_within_envelope` at the install-time + delegation-time seams.
- `benten-sync` consumes the `RateLimitPolicy` plug at the Atrium boundary for per-peer bandwidth accounting.

**No cycles.** `benten-caps` sits between `benten-graph` (below it) and `benten-eval` / `benten-engine` / `benten-platform-foundation` (above it). It does not depend on `benten-eval` (which would invert the layering). The prod edge to `benten-platform-foundation` is asymmetric (Cargo-cycle-safe: foundation's edge back to caps stays dev-only). Integration tests that need a full `Engine` do so as dev-deps only, and the dev-cycle is safe because test binaries link separately from the lib crate graph.

## 4. Public API surface

- **Context types:** `WriteContext`, `ReadContext`, `PendingOp`, `WriteAuthority` (re-exported from `benten-core`).
- **Concrete policies:** `NoAuthBackend`, `LegacyUcanStubBackend`, `GrantBackedPolicy`, `UcanGroundedPolicy` (native-only), `UCANBackend` (native-only, the *backend*, not a `CapabilityPolicy` itself ‚Äî composed into `UcanGroundedPolicy`).

## 7. Thin-engine + composable-graph philosophy check

The crate's load-bearing posture is: **capability is policy, the trait is the contract, backends are pluggable** (CLAUDE.md baked-in #7). Phase 4-Foundation R5 + R6-FP extended that posture into the plugin trust composition (CLAUDE.md baked-in #18 three-layer model) ‚Äî the chain-validator + envelope-checker + scope-deriver compose ON TOP of the existing `CapabilityPolicy` surface; they do NOT introduce a fourth policy impl. The places where the philosophy is honoured + the places where there's friction:

### Well-respected examples

1. **Trait surface is principal-agnostic.** `WriteContext` carries `actor_cid: Option<Cid>` + `actor_hint: Option<String>` + `device_cid: Option<Cid>` + the `pending_ops` batch + the `authority: WriteAuthority` axis. No backend-specific shape is baked in. UCAN-specific principal types (DIDs, signed envelopes, attestation envelopes) stay over the `benten-id` line; plugin-manifest types stay over the `benten-platform-foundation` line; this crate sees them only as opaque `Cid` / `String` / typed `Ucan` / `Did` values.
2. **Object safety is preserved.** Every trait method has a concrete-return signature. No `where Self: Sized` defaults that take `self` by value. `Arc<dyn CapabilityPolicy>` / `Arc<dyn GrantReader>` / `Arc<dyn RateLimitPolicy>` boxing all work ‚Äî integration tests routinely do this.
3. **GrantReader is a separate trait, not a method on `CapabilityPolicy`.** The shape of storage-read access has nothing to do with the shape of policy decisions; folding them would force every policy to think about storage. The trait split is the layering boundary that keeps `benten-caps` from taking a direct dep on `benten-graph` for the policy types themselves. The Phase-4-Foundation R1 principal-aware extension (`has_unrevoked_grant_for_scope_and_actor`) ships with a backward-compat default impl that collapses to the scope-only path ‚Äî every external `GrantReader` continues to compile, only `BackendGrantReader` overrides to filter on `grantee`.
4. **RateLimitPolicy is a separate trait, not a method on `CapabilityPolicy`.** The Atrium-boundary bandwidth budget is independently configurable from the capability gate. Composing via `Arc<dyn RateLimitPolicy>` on `UCANBackend` avoids growing another generic parameter.
5. **`#[non_exhaustive]` on `PendingOp` + `CapError`.** Phase-2b SANDBOX `HostFunctionCall` + Phase-3 UCAN-specific error variants land as minor bumps; external `match` expressions must include `_ =>`. The forward-compat discipline mirrors `ErrorCode` and `GraphError`.
6. **Dual durable-grant-store seam intentionally documented.** `backends/ucan.rs` carries a load-bearing module-doc paragraph explaining why the raw-KV `g14b:grant:<cid>` store and the Node-encoded `system:CapabilityGrant` store coexist. A "unify the stores" PR is rejected by reference to this paragraph.
7. **Plugin-trust composition does NOT introduce a fourth `CapabilityPolicy` impl.** G24-D + G24-D-FP-2 + G27-D ship as composable building blocks consumed at the engine surface (the engine wires the chain validator into `Engine::install_plugin` + the audience-side envelope check into the write-path). The trait surface is unchanged; the new modules are pure functions + trait abstractions that compose with the existing `CapabilityPolicy`. This honors the "trait is the contract" posture.
8. **`SharesPolicyView` + `ManifestEnvelopeLookup` + `UserDidRegistry` are minimal abstract traits.** They define the absolute minimum surface this crate needs to reason about manifest envelopes + user-roots, with concrete impls living over the `benten-platform-foundation` boundary. The prod-edge to platform-foundation exists (G27-D / batch-3 onwards) but stays Cargo-cycle-safe (foundation's reverse edge is dev-only).

### Frictions / surface concerns to flag

1. **`UcanGroundedPolicy` scope-string surface still keys off `cap:typed:*` namespace.** The composed-policy `check_write` literally string-prefix-checks `ctx.scope` for `"cap:typed:"` and routes to `typed_cap_permitted_by_proof` only for that namespace. Phase-4-Foundation R1-FP G22-FP-2 added audience binding via `principal_did_from_context` (lines 103+) ‚Äî sourcing the principal from `WriteContext::actor_hint` when it carries a `did:key:` prefix ‚Äî but the full extension to arbitrary scope strings still requires the `WriteContext::actor_cid ‚Üí Did` resolution helper at the engine boundary (cap-r1-16 seam). Once that lands, the `cap:typed:*` prefix check becomes a non-special-case routing decision.

2. **`DEFAULT_NOW_SECS = 0` sentinel is a load-bearing inversion that depends on a discipline.** The G16-B-B-rest fail-closed inversion in `UcanGroundedPolicy` keys off `now_secs == 0` AND `chain_has_time_bounds`. The sentinel works because production callers MUST inject a real wallclock via `with_now_for_test` or (eventually) `WriteContext::now` threading. Until that threading lands (`phase-3-backlog ¬ß2.3 (i)`), the sentinel is the only operator-visible signal that "no clock was injected." The right cleanup is `WriteContext::now` threading; the right interim defense is the typed `UcanClockNotInjected` error that surfaces the misconfiguration loudly.

3. **`GrantBackedPolicy` scope derivation now has TWO paths.** Phase-1 derives `format!("store:{label}:write")` from `ctx.label`; G27-B introduced the explicit `ctx.scope` override that SHORT-CIRCUITS the per-op derivation. The override path expresses non-CRUD plugin-manifest grammar (`private:<plugin_did>:*`, `requires:<plugin_did>:<path>`, `shares:<plugin_did>:<path>`) without expanding label derivation. **Outstanding G27-B mini-review MINOR (`r5g27b-mr-2`):** the override SHORT-CIRCUITS per-op derivation entirely, including the `system:*` skip + read-before-delete idempotent-miss semantics. The SOLE call site that currently sets `ctx.scope` with potentially-populated `pending_ops` is `Engine::wait_resume_action` at `engine_wait.rs:884` (`pending_ops` empty in practice); future call sites that combine explicit `ctx.scope` with a mixed `pending_ops` batch (e.g., regular op + `system:*` op) MUST audit whether the higher-grain override is the intended semantic ‚Äî the policy no longer enforces caps per-op when both are set. See `tests/grant_backed_policy_scope_overrides_pending_op_label_derivation.rs` (referenced inline; if absent at HEAD this is the registered v1-window seam).

4. **¬ß13.11 root-cause structural lesson ‚Äî `BackendGrantReader::revoked_scopes` matched by scope-string equality (CLOSED at PR #199; structural lesson preserved).** The Phase-3 closure surfaced that the production gap (silent fail-OPEN on `revokeCapability(grantCid, actor)`) was structurally enabled by the policy reading revocations *by scope string* while the napi surface passed the grant CID *as* the scope. PR #199 shipped `Engine::revoke_capability_by_grant_cid` ‚Äî resolves the CID to a Node, extracts the `scope: Text` property, routes through the existing `(actor, scope)` revoke path. The structural lesson on the `benten-caps` side is preserved: `GrantReader::has_unrevoked_grant_for_scope(scope: &str)` is *correct* but silent on **how the engine builds the scope-string**. G27-C extended `benten-id` with a sibling `GrantReader::has_unrevoked_grant_for_grant_cid(cid: &Cid)` trait (CID-keyed companion); production callers that don't know the scope-string layout now have a typed handle. This makes the CID-keyed path the canonical typed API even for callers like `Engine::revoke_capability_by_grant_cid`.

5. **`GrantBackedPolicy::wildcard_variants` is an O(2^N) enumeration.** For an N-segment required scope, the policy enumerates up to 2^N candidate parent-scope spellings and queries each through `has_unrevoked_grant_for_scope`. Bounded to N ‚â§ 6 (bails to `[required, "*"]` above that) but still up to 32 reader hits per write check. Cheap when N is 3 (the Phase-1 CRUD case) and the reader is in-process; harder to defend when N grows and the reader queries a remote KV. The right cleanup is for `GrantReader` to expose a wildcard-aware query the policy calls once (matching the `has_unrevoked_grant_for_any` precedent). Surfaced as a Phase-4-Meta carry.

6. **Plugin-trust composition layering = 3 surfaces (issuance / audience / chain).** G24-D `plugin_delegation` owns the **issuance**-side single-step gate (source plugin issues a UCAN to target plugin ‚Äî does source's `shares` policy permit?). G27-D `manifest_scope::check_scope_within_envelope` owns the **audience**-side write-path check (audience plugin's write fires with a concrete cap scope ‚Äî is that scope within source manifest's envelope?). G24-D-FP-2 `manifest_envelope_chain_validation` owns the **full-chain** walk (compose the issuance gate across every hop + assert user-DID root anchor + chain integrity). The three checks operate at different points in the cap lifecycle; the audience-side + chain-side checks both consume `SharesPolicyView` but at different grains. Frictionful: the symmetric pair (issuance vs audience) requires keeping two implementations of "is this scope within this manifest's envelope?" in sync ‚Äî the `manifest_scope` doc names this seam explicitly. Both delegate to `SharesPolicy::permits_delegation` (the source-of-truth defined in `benten-platform-foundation`), which is the right shape ‚Äî the two surface entry points + one source-of-truth.

7. **No durable-vs-ephemeral assumption baked in.** `CapabilityPolicy` doesn't presume the underlying store is durable. `NoAuthBackend` has no store at all; `GrantBackedPolicy` consults whatever `GrantReader` is injected (could be redb, could be in-memory `BTreeMap` for tests); `UCANBackend<B: GraphBackend>` is parametric over the backend shape. Good.

8. **No coupling to specific Atrium/sync shapes at the policy trait.** `WriteContext::device_cid` is an `Option<Cid>` ‚Äî the policy can dispatch on it but doesn't require it. `WriteAuthority::SyncReplica` is a variant on the re-exported core enum, so the policy can match on "this write arrived via sync" without this crate caring about iroh / Loro / MST details. The `RateLimitPolicy::check_peer_bandwidth` surface takes `peer: &str` so the rate-limit plug doesn't need to know about iroh's `NodeId` shape either. Good ‚Äî `benten-sync` is the only crate that translates from network identity to `peer: &str`.

9. **Compromise #26 ‚Äî manifest envelope re-check at write time is PARTIALLY CLOSED.** Phase-4-Foundation R4b-FP-1 shipped Seam 3 (the `apply_atrium_merge` envelope-recheck inline call wiring); the production `ManifestEnvelopeRechecker` adapter is named for Phase-4-Meta ¬ß4.36. At HEAD the chain-validator + audience-side envelope check fire at install-time + at the cap-policy boundary, but the engine-side adapter that re-checks the envelope on every inbound sync-merge row remains a v1-assessment-window item. Future agents proposing to retire the partial-close framing must reference ¬ß4.36 directly.

## 9. Open questions / unresolved internals

1. **When does `WriteContext::now` threading land?** Named at `phase-3-backlog ¬ß2.3 (i)` as the v1-assessment-window deliverable that retires the `DEFAULT_NOW_SECS = 0` sentinel. Two open shape questions: does `now` go on `WriteContext` or on the policy itself? Does the evaluator inject a per-CALL `now` or a per-batch `now`?

2. **Should `wildcard_variants` move into `GrantReader`?** Per ¬ß7 item 5 ‚Äî O(2^N) enumeration on the policy side could collapse into a single wildcard-aware reader call. Open shape: does this break the "reader is opaque to wildcards" property that lets simple test fixtures implement just one method?

3. **Is there a future split between policy-with-grants and policy-with-chains?** Today `UcanGroundedPolicy` composes `GrantBackedPolicy` + `UCANBackend`. The composition is hard-coded to "grant-first, chain-second" with the `cap:typed:*` prefix as the routing key. A more layered shape might be `CapabilityPolicy` impls that can be *stacked* (an `Or<P1, P2>` combinator, or a per-scope-namespace dispatch). Open question: is the stacked shape worth the abstraction cost, or does the current "one composed policy per namespace family" hold up through Phase 4-Meta?

4. **`InMemoryRateLimitPolicy` is the only concrete `RateLimitPolicy`.** Per the trait module doc, "distributed token-bucket across Atrium peers" lands at G14-D / G16. Open question: is a single concrete plug enough through v1, or does the Atrium boundary need a distributed-counter shape before plugin-marketplace traffic patterns (Phase 6+) make in-memory accounting laughable? Probably yes for v1; flag for Phase 6 dispatch.

5. **`UCANBackend::iter_installed_proofs` silently skips decode failures.** The source comment names this as a future-hardening axis: "tighten to surfacing a typed `CapError::BackendStorage` when decode fails (would require distinguishing real corruption from forward-compat envelope shapes)." Open question: when forward-compat envelope shapes start arriving (Phase 4 plugin signing? Phase 6 cross-network UCANs?), what's the right disposition for "I can't decode this entry"? Skip-and-log? Fail-closed? Treat as revoked?

6. **`LegacyUcanStubBackend` retention timeline.** Renamed at G21-T2 to disambiguate from `UCANBackend`. The error-routing contract (`ON_ERROR` not `ON_DENIED`) is what's load-bearing; the stub itself is dead weight that survives because some Phase-1 / Phase-2a tests still import it. Open question: when can these tests be retired or migrated to the durable backend so the stub goes away?

7. **`principal_did_from_context` source-of-truth.** The G22-FP-2 helper sources the principal `Did` from `WriteContext::actor_hint` (looking for `did:key:` prefix + round-trip). This works because Phase-4-Foundation R1 didn't ship the `actor_cid ‚Üí Did` resolution helper at the engine boundary. Open question: when cap-r1-16 closes that seam, does `principal_did_from_context` move to consult `actor_cid` first (and `actor_hint` becomes a Phase-1 / test-fixture fallback)?

8. **Audience-side vs issuance-side envelope checks ‚Äî keeping them in sync.** Per ¬ß7 item 6: `plugin_delegation::check_delegation_within_envelope` (issuance) and `manifest_scope::check_scope_within_envelope` (audience) both delegate to `SharesPolicy::permits_delegation` (source of truth in `benten-platform-foundation`). The seam works at HEAD; the open question is whether a future divergence (e.g. issuance-time grace period vs audience-time strict check) ever surfaces ‚Äî at which point the SOLE source-of-truth shape would need a structural decision.
