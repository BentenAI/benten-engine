# benten-napi — Internals

Plain-English deep-dive of `bindings/napi/`, the napi-rs v3 Node.js (and `wasm32-unknown-unknown`) binding layer that exposes the Benten Engine to TypeScript callers. Audience: a developer landing in this crate fresh and trying to find the load-bearing seams. Read-only audit. Snapshot HEAD: `c589ffe` (post `phase-4-foundation-close` tag).

---

## 1. What this binding does

`benten-napi` is the FFI seam between the Rust workspace's [`benten_engine::Engine`](crates/benten-engine/INTERNALS.md) (and a handful of supporting crates — `benten-core`, `benten-caps`, `benten-id`, `benten-eval`, `benten-ivm`, `benten-dsl-compiler`, `benten-dev`) and the JavaScript / TypeScript runtime that the `@benten/engine` npm package wraps. Every method a JS caller invokes on the high-level `@benten/engine` `Engine` class ultimately lands on a `#[napi]` method here. Values cross the boundary as `serde_json::Value`; CIDs cross as base32-multibase strings (prefix `b`); raw bytes cross as `Buffer` / `Uint8Array`.

The crate is unusual in that it produces TWO artifacts from one source tree, driven by `crate-type = ["cdylib", "rlib"]` plus a `napi-export` feature gate ([Cargo.toml:14-39](bindings/napi/Cargo.toml#L14-L39)). The `cdylib` is the `.node` consumer-facing binary that Node.js / Bun / Deno `dlopen`s; the `rlib` is a pure-Rust library the in-process integration tests under `bindings/napi/tests/` link against without resolving napi extern symbols. The `napi-export` feature pulls `dep:napi` + `dep:napi-derive` for the cdylib; disabling it yields the rlib path. This duality is what lets a single `tests/input_validation.rs` test binary exercise the same code paths the Node.js cdylib does without spinning up a JS runtime — the test runs entirely inside `cargo test`.

The binding covers the **Phase-1 / Phase-2 / Phase-3 full-peer engine + sync surface** (shape (a) per CLAUDE.md baked-in #17). Phase-4-Foundation's admin UI / plugin manifest / schema-compiler / materializer surfaces are reached via the **wasm32 thin-client protocol** (shape (b) / shape (c)), NOT via napi — this is the intentional architectural scope boundary documented at [lib.rs:11-33](bindings/napi/src/lib.rs#L11-L33), not staleness.

## 2. Where it sits in the dep + deployment chain

### Inputs (workspace)
- `benten-engine` (with `iteration-budget-test-grade` feature) — the public `Engine` alias the cdylib wraps.
- `benten-core` — `Node`, `Edge`, `Value`, `Cid` round-trip types.
- `benten-caps` — `PolicyKind` enum substrate.
- `benten-errors` — stable catalog codes embedded in the JSON error envelope.
- `benten-dsl-compiler` — typed `Diagnostic` shape the napi `DevServer` surfaces on bad-DSL input.
- `benten-ivm` — `Strategy` enum referenced by the user-view spec parser.
- Native-only (`cfg(not(target_arch = "wasm32"))`): `benten-dev` (devserver scaffolding), `benten-id` (identity primitives + Ed25519 ops), `tokio` (singleton runtime that drives Atrium async).

### Inputs (external)
- `napi` (`workspace`, `napi4` + `serde-json` features, `optional = true` behind `napi-export`)
- `napi-derive` (same gating)
- `blake3` (synthetic-CID derivation for friendly principal strings via `parse_actor_cid_or_derive`)
- `data-encoding` (base64 codec for `SuspensionBridge` handle bytes)
- `serde_json` (the load-bearing wire format at the JS boundary)
- `tempfile` (optional, behind `in-process-test`)

### Build-deps
- `napi-build = "2"` — drives the `.node` glue code generation via [`build.rs`](bindings/napi/build.rs).

### Workspace consumers (out)
The npm package [`packages/engine/`](packages/engine) consumes the cdylib as its compiled back-end (`@benten/engine-native` per [`package.json`](bindings/napi/package.json#L2)). The `index.d.ts` (generated by napi-rs at build time) defines the TS-side typed surface. Adjacent npm packages — `@benten/engine-devserver`, the admin-UI tooling, vitest fixtures — consume `@benten/engine` not the cdylib directly.

### External consumers
Eventual app authors using `@benten/engine` for full-peer Node.js apps (CLI tools, Tauri shells per CLAUDE.md #17 shape (c), Phase-9+ Benten Runtime hosts).

### Three-deployment-shape positioning per CLAUDE.md baked-in #17
- **Shape (a) full peer** — the native `cdylib` (`benten-napi.<triple>.node`). Durable redb storage, full Atrium sync (iroh + Loro CRDT), SANDBOX runtime (wasmtime), `benten-id` cryptographic operations. This is the binding's primary deployment shape.
- **Shape (b) thin compute surface** — the same source tree compiled to `wasm32-unknown-unknown`. The `napi_surface::Engine` class itself is **`cfg(not(target_arch = "wasm32"))`**-gated ([lib.rs:204-221](bindings/napi/src/lib.rs#L204-L221)) so the wasm32 bundle exports ONLY the in-memory `wasm_browser::BrowserManifestStore` + the `browserRuntimeAvailable()` probe + the IndexedDB-backed thin-client cache surfaces. Browser tabs / edge workers participate as authenticated views into a full peer, not as engines themselves.
- **Shape (c) embedded webview** — Phase-4-Foundation ships Tauri 2.x; the Tauri shell loads the same `wasm32-unknown-unknown` bundle as shape (b) talking to an embedded shape (a) full peer over in-process IPC. The napi binding itself does NOT participate in shape (c) — it's loaded only by the native shell, not the webview.

### Strict architectural rules
- **`crate-type = ["cdylib", "rlib"]`** is load-bearing: the rlib half is what lets `tests/input_validation.rs` link without resolving napi externs. The `napi-export` feature gates the napi-rs surface so the rlib path produces a clean pure-Rust library.
- **`wasm32-unknown-unknown` build refusal for shape-(a) surfaces.** The `napi_surface` module, `wait` / `stream` / `subscribe` / `devserver` / `identity` / `atrium` adapter modules are all `cfg(not(target_arch = "wasm32"))`-gated. The wasm32 build deliberately compiles ONLY the thin-client cache + presence-probe surfaces.
- **Class B β plugin-engine seam (CLAUDE.md baked-in #18).** The napi binding does NOT expose `Engine::read_node_as` directly — `Engine::call_as` carries the active principal through the evaluator; reads inside the walk thread through `read_node_as` engine-internally, never crossing the napi boundary. The napi `Engine::get_node` exposes the user-facing default-no-principal path only ([lib.rs:329](bindings/napi/src/lib.rs#L329)). Plugin authors author graph nodes; the evaluator is the only caller of `_as`.
- **Phase-4-Foundation scope boundary.** Per [lib.rs:11-33](bindings/napi/src/lib.rs#L11-L33), the napi binding does NOT mirror plugin manifest / plugin library / plugin lifecycle / schema compiler / materializer / module ecosystem surfaces. Those reach JS via the wasm32 thin-client protocol — duplicating them through napi would break the heterogeneity contract.

## 4. Public surface highlights

The generated [`index.d.ts`](bindings/napi/index.d.ts) catalogues the full TS-facing surface. The load-bearing exports are six `#[napi]` classes + four free functions + one string enum:

### 4a. The `Engine` class
Wraps `Arc<benten_engine::Engine>`. ~70 methods covering CRUD, registration, call/trace, capabilities, views, WAIT/STREAM/SUBSCRIBE/EMIT, snapshot, module install, and diagnostics. Defined at [lib.rs:261-1671](bindings/napi/src/lib.rs#L261-L1671). Construction:
- `new Engine(path: string)` — opens or creates an engine against a local redb file. The `":memory:"` sentinel routes to a transient redb store.
- `Engine.openWithPolicy(path, policy: PolicyKind)` — `static factory` with explicit `PolicyKind`.
- `Engine.fromSnapshotBlob(bytes: Buffer)` — `static factory` for snapshot-blob handoff (native-only; wasm32 arm fires `E_BACKEND_READ_ONLY`).
- `engine.atrium(config: AtriumConfig)` — factory method returning a `JsAtrium` handle (Pattern B-prime per D1 ratification; deferred-bind, the actual iroh `Endpoint` bind happens on `.join()`).

Notable methods:
- `engine.call(handlerId, op, input)` and `engine.callAs(handlerId, op, input, actor)` — handler dispatch. The `actor` arg uses `parse_actor_cid_or_derive` so friendly principal strings (`"alice"`) hash to deterministic synthetic CIDs while real base32 CIDs pass through verbatim.
- `engine.typedCall(opName, input)` — Phase-3 G21-T2. Routes through `Engine::dispatch_typed_call_public` for the 10 typed-CALL ops (Ed25519 sign/verify, keypair gen, blake3 hash, multibase, DID resolve, UCAN chain validate, VC verify). Surfaces typed `E_TYPED_CALL_*` errors.
- `engine.delegateCapability(source, pluginDid, caps)` — Phase-4-Foundation runtime delegation envelope seam ([lib.rs:737](bindings/napi/src/lib.rs#L737)). Drives `Engine::caps().delegate_capability(...)`.
- `engine.registerSubgraphReplace(spec)` — R6FP-tail Instance 10 closure. Returns full JSON `{handlerId, cid, previousCid, chainDepth, versionTag, replaced}` so JS callers can distinguish first-registration / idempotent re-registration / true replace.
- `engine.testingSetIterationBudget(budget)` — narrowly gated on the dedicated `iteration-budget-test-grade` feature so the production cdylib has the surface without pulling the broader `test-helpers` API.

Cross-link to [`crates/benten-engine/INTERNALS.md`](crates/benten-engine/INTERNALS.md) §4 for the substrate behind each method.

### 4b. Subscription / Stream / EMIT JS handles
- `SubscriptionJs` ([lib.rs:1675-1753](bindings/napi/src/lib.rs#L1675-L1753)) — holds the underlying `benten_engine::Subscription` behind a `Mutex` for the lifetime of the JS-side reference. `isActive` / `pattern` / `maxDeliveredSeq` (returns `i64` for safe JS-number domain) / `unsubscribe`. The Mutex+holds-Subscription shape is the cr-w8c-fp-1 fix-pass closure — the prior return-as-JSON shape dropped the Subscription at end-of-method-scope, releasing the `napi::ThreadsafeFunction` Arc so JS callbacks could never fire.
- `EmitSubscriptionJs` ([lib.rs:1779-1831](bindings/napi/src/lib.rs#L1779-L1831)) — same shape minus `pattern` / `maxDeliveredSeq` (EMIT events are broadcast, not seq-tracked). `channel()` accessor.
- `StreamHandleJs` ([lib.rs:1854-1969](bindings/napi/src/lib.rs#L1854-L1969)) — JS-side stream handle wrapping `benten_engine::StreamHandle` in `Mutex<Option<...>>`. `next()` returns `Buffer | null` (null = end-of-stream); `close()` is idempotent; `isDrained` / `seqSoFar` (i64) / `requiresExplicitClose` reflect the underlying handle state.

### 4c. DevServer + ReloadSubscriber
`DevServer` (native-only, [devserver.rs](bindings/napi/src/devserver.rs)) — start/stop/register/replace lifecycle plus a `ReloadSubscriberJs` factory for hot-reload event drains.

### 4d. JsAtrium + AtriumConfig + DeviceAttestationDeclaration
[`atrium.rs`](bindings/napi/src/atrium.rs) hosts `JsAtrium`, the typed config struct `AtriumConfig` ([atrium.rs:102](bindings/napi/src/atrium.rs#L102)), the `DeviceAttestationDeclaration` and `CapabilityClaim` typed napi objects, plus `JsAtrium`'s 14 methods covering join/leave, trust-roster ops, declared-attestation registration, and the engine-side `AtriumHandle` drives.

### 4e. JsKeypair + JsDeviceAttestation
[`identity.rs`](bindings/napi/src/identity.rs) exposes Ed25519 primitives for the full-peer side. The browser thin-client declares the same shape via `packages/engine/src/identity.ts` but routes signing through to a full peer at handshake time.

## 5. How it's used (concrete call paths)

### 5a. `engine.createNode(labels, properties)` → Rust → backend
1. JS calls `nativeEngine.createNode(["Post"], {title: "Hello"})`.
2. napi-rs marshals the args into `(Vec<String>, serde_json::Value)`.
3. [`napi_surface::Engine::create_node`](bindings/napi/src/lib.rs#L316) calls [`json_to_props`](bindings/napi/src/node.rs#L80) which walks the JSON tree, charges every leaf against the `ByteBudget` accumulator (depth ≤ 128, map keys ≤ 10K, single string ≤ 1 MiB, aggregate ≤ 16 MiB), and produces a `BTreeMap<String, Value>`.
4. `CoreNode::new(labels, props)` builds the in-memory Node.
5. `self.inner.create_node(&node)` lands on [`benten_engine::Engine::create_node`](crates/benten-engine/INTERNALS.md) which opens a single backend transaction, fires the cap-policy pre-write hook, writes the canonical-bytes CBOR, computes the BLAKE3-derived CID, and commits.
6. The returned `Cid` is rendered via `cid.to_base32()` and crosses back to JS as a `"bafyr4i..."` string.
7. On any error (cap-deny, input-limit, backend-fail), the error path lands in [`engine_err`](bindings/napi/src/error.rs#L42) which JSON-encodes `{code: "E_*", message: "...", fields: {...}}` into a `napi::Error`. TS-side [`mapNativeError`](packages/engine/src/errors.ts) JSON-parses it and populates the typed `BentenError` subclass via the generated `CODE_TO_CTOR_GENERATED` table.

### 5b. `engine.onChangeAs(pattern, actor, callback)` → cross-thread delivery
1. JS calls `engine.onChangeAs("post:*", "alice", (seq, payload) => { ... })`.
2. napi-rs converts the JS callback into a `Function<'_, FnArgs<(u32, Buffer)>, ()>` argument.
3. [`napi_surface::Engine::on_change_as`](bindings/napi/src/lib.rs#L1310) parses the cursor JSON via [`parse_cursor`](bindings/napi/src/subscribe.rs#L126), derives the actor CID via `parse_actor_cid_or_derive`, and calls [`on_change_as_adapter`](bindings/napi/src/subscribe.rs#L197).
4. The adapter wraps the JS callback in a `napi::ThreadsafeFunction` via [`build_on_change_tsfn`](bindings/napi/src/subscribe.rs#L70) — the `FnArgs<(u32, Buffer)>` wrapper is load-bearing here (without it napi-rs delivers the tuple as a single Array, breaking the documented `(seq, payload)` callback contract).
5. `engine.on_change_as_with_cursor(pattern, cursor, engine_cb, actor)` registers a Phase-4-Foundation Option-D `CapRecheckFn` closure (returning `CapRecheckOutcome::Cancel` on whole-actor revoke, `Drop` on per-event cap-recheck deny, `Deliver` otherwise per [`crates/benten-engine/src/engine_subscribe.rs`](crates/benten-engine/INTERNALS.md)).
6. When a WRITE commits on any thread, the `ChangeBroadcast` fan-out walks subscribers synchronously; the engine-side closure projects `(seq32, payload_bytes)` and calls `tsfn.call(...)` with `NonBlocking` semantics — napi-rs enqueues onto the libuv main loop.
7. The TSFN's build_callback translates the tuple into `(JS number, JS Buffer)` and invokes the JS callback on Node's main thread.
8. The returned `Subscription` is wrapped in `SubscriptionJs::from_inner(sub)` and exposed to JS. Drop on the JS handle unregisters the on-change entry and releases the TSFN Arc.

## 6. Compromises + cross-language discipline

### 6a. ErrorCode 4-surface mirror (`.addl/dispatch-conventions.md` §3.5g)
The Rust `benten_errors::ErrorCode` catalog (168 throwable variants at HEAD `c589ffe`) is the source of truth. The TS-side mirror at `packages/engine/src/errors.generated.ts` is regenerated from `docs/ERROR-CATALOG.md` via [`scripts/codegen-errors.ts`](scripts/codegen-errors.ts). The §3.5g rule-mirror discipline requires any ErrorCode mint atomically touches four surfaces in the same commit: (1) `crates/benten-errors/src/lib.rs` enum + `as_str`/`from_str` arms, (2) `crates/benten-errors/tests/stable_shape.rs::ALL_CATALOG_VARIANTS`, (3) `docs/ERROR-CATALOG.md` narrative entry, (4) `packages/engine/src/errors.generated.ts`. The drift detector at [`scripts/drift-detect.ts`](scripts/drift-detect.ts) (CI workflow [`.github/workflows/drift-detect.yml`](.github/workflows/drift-detect.yml)) compares all four and fires on divergence.

The napi binding consumes the Rust side — every `EngineError` / `CoreError` variant flows through the JSON envelope formatter in [`error_envelope.rs`](bindings/napi/src/error_envelope.rs) and the catalog code crosses to JS as the literal `"code": "E_*"` field. The TS `BentenError.context` accessor is populated from the `"fields"` half when `EngineError::context_json` returns `Some(...)`.

### 6b. HostError wire format excludes CID bytes
The SANDBOX HostError surface deliberately does NOT carry CID bytes across the boundary — per Compromise #3 + Compromise #4 the napi error envelope carries the typed `ErrorCode` (Compromise #3 closure: `ErrorCode` lives in `benten-core`) + Display message + optional `fields` only; CID-bytes are deliberately excluded from the wire format per the SANDBOX-surface wave-8h discipline (Compromise #4 closure: WASM runtime is compile-check only). The metric-propagation gap is Compromise #17: `engine.describeSandboxNode(handlerId, nodeId)` synthesizes the `SandboxNodeDescription` shape client-side from the registered subgraph spec the wrapper cached and returns the literal `"unknown"` sentinel for `fuelConsumedHighWater` / `lastInvocationMs` (deliberate type-level signal that the value is structurally not tracked yet — see [`sandbox.rs:98-137`](bindings/napi/src/sandbox.rs#L98-L137)).

### 6c. wasm32-unknown-unknown REFUSAL is deliberate
The `napi_surface::Engine` class is `cfg(not(target_arch = "wasm32"))`-gated ([lib.rs:204-221](bindings/napi/src/lib.rs#L204-L221)) per CLAUDE.md baked-in #17. The wasm32 bundle is a thin compute surface: stateless reads through authenticated thin-client subscription protocol against a full peer, not a full engine. The wasm32 target compiles ONLY the in-memory `BrowserManifestStore` + IndexedDB cache surfaces + presence probes. Future agent proposals to ship Loro / iroh / `Engine::call` in the wasm32-unknown-unknown bundle must be rejected with reference to CLAUDE.md #17.

### 6d. 3-rung baked-in-#17 defense (R6 fix-pass Wave B closure)
[`tests/feature_graph_closure_no_test_helpers_in_production.rs`](bindings/napi/tests/feature_graph_closure_no_test_helpers_in_production.rs) walks `bindings/napi/Cargo.toml`'s feature graph to assert the production cdylib's default-feature closure does NOT transitively activate `benten-eval/test-helpers` or `benten-engine/test-helpers`. Closes br-r6-r1-3 MAJOR + R6 R1 browser-wasm-bundle lens corroboration.

### 6e. Rebake-on-Rust-shape-change discipline

### 6g. Safe-4 #704 closure + #1202/#1203 AsyncTask migration HARD-STOP (refinement-audit-2026-05 napi crate-drain lane)

**#704 — CLOSED (this lane).** `JsAtrium::is_active` previously held the sync `state` mutex across the `handle.is_active()` call (`let state = self.state.lock()...; if let Some(handle) = state.engine_atrium.as_ref() { handle.is_active() }`). `AtriumHandle::is_active` is a bare atomic load today so this was presently safe, but the lock-held-across-handle-call SHAPE is a latent re-entrancy / deadlock hazard if `is_active` ever evolves to acquire an engine-side `tokio::sync::Mutex` or becomes `async`. A runtime concurrency-stress closure-pin landed as the `#[cfg(test)] mod is_active_lock_scope_704_tests` in [`atrium.rs`](bindings/napi/src/atrium.rs) — it hammers `is_active` from 8 threads while 4 threads mutate `state` via `trust_peer` + 1 flips the joined flag, and would FAIL (deadlock → harness timeout) on a revert combined with a lock-acquiring handle dispatch. That `#[cfg(test)]` module does NOT affect the cdylib / `cargo check` / `cargo clippy` surface (all green) but it cannot RUN under scoped-pre-flight: any test compiling `JsAtrium` requires the `napi-export` feature whose Node-runtime extern symbols (`_napi_register_module_v1` / `_napi_delete_reference` / `_napi_wrap`) are only provided by the build-harness (the CLAUDE.md §0 documented pre-existing limitation — the same reason the `tests/atrium_join_production_mode_1187.rs` pin tests engine-side types only and the `in-process-test` tests reach napi behavior via `benten_engine::Engine`). The pin RUNS the moment the dedicated build-harness-enabled wave (below) compiles it.

**#1202 / #1203 (META #744 AsyncTask migration PR-A/PR-B) — HARD-STOP / SURFACED.** Re-verified at HEAD `6ed80b2f`: zero `AsyncTask`/Promise-returning `#[napi]` methods across the binding; 12 `block_on` calls remain (sub-issues #648/#688/#704/#652/#716 are CLOSED only via C-3 dedup-consolidation into these umbrellas — the migration work itself has NOT landed; the #1282 ST-NAPI lane landed only the orthogonal #1187 production-mode fix + #1206-partial cdylib closures).
